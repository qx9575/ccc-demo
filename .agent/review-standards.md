# 代码审查标准

本文档定义了 Reviewer Agent 在审查代码时遵循的标准。

## 1. 代码质量 (code_quality)

### 1.1 代码风格
- 遵循项目既定的代码风格规范
- 命名规范：变量、函数、类命名清晰有意义
- 缩进和格式一致
- 避免过长的函数和文件

### 1.2 可读性
- 代码逻辑清晰
- 适当的注释
- 避免过于复杂的嵌套
- 使用有意义的变量名

### 1.3 可维护性
- 避免硬编码
- 适当的抽象层次
- 避免重复代码
- 模块化设计

## 2. 测试覆盖 (test_coverage)

### 2.1 单元测试
- 每个函数都有对应的测试
- 测试覆盖正常路径和边界条件
- 测试覆盖错误处理路径

### 2.2 测试质量
- 测试用例独立
- 测试名称描述清晰
- 断言充分且有意义

## 3. 安全性 (security)

### 3.1 输入验证
- 所有外部输入都经过验证
- 防止 SQL 注入
- 防止 XSS 攻击
- 防止命令注入

### 3.2 数据保护
- 敏感数据加密存储
- 不在日志中记录敏感信息
- 不在代码中硬编码密钥

### 3.3 权限控制
- 适当的权限检查
- 最小权限原则

## 4. 性能 (performance)

### 4.1 算法效率
- 选择合适的数据结构和算法
- 避免不必要的循环和递归
- 注意时间复杂度和空间复杂度

### 4.2 资源使用
- 避免内存泄漏
- 合理使用数据库连接
- 适当的缓存策略

## 5. 文档 (documentation)

### 5.1 代码注释
- 函数注释说明用途、参数、返回值
- 复杂逻辑有解释性注释
- TODO 注释标记待办事项

### 5.2 API 文档
- API 端点有清晰描述
- 请求/响应格式有说明
- 错误码有定义

## 审查结果判定

### approved (通过)
满足以上所有必要标准，无重大问题。

### rejected (驳回)
存在以下任一情况：
- 安全漏洞
- 功能不完整
- 测试失败或覆盖不足
- 代码质量严重不达标
- 性能问题严重

### suggestions (建议修改)
满足必要标准，但有改进空间：
- 代码可读性可提升
- 性能可优化
- 文档可完善
